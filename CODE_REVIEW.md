# 代码审查流程文档

## 📋 概述

本文档定义了 yujia-homework-cat 项目的代码审查流程和标准。

---

## 🔍 审查流程

### 1. 自检阶段

在提交代码之前，开发者必须完成以下检查：

- [ ] 代码符合 `CODE_STANDARDS.md` 中的命名规范
- [ ] 代码通过 ESLint 检查（无 Error 级别问题）
- [ ] 代码使用 Prettier 格式化
- [ ] 逻辑正确，无明显的 Bug
- [ ] 注释清晰，复杂逻辑有说明
- [ ] 没有引入不必要的依赖

**运行命令**：
```bash
# 代码检查
npm run lint

# 代码格式化
npm run format
```

### 2. 提交审查

- 使用规范的提交信息格式（参考下方）
- 推送到功能分支或开发分支
- 创建 Pull Request（如果有多人协作）

**提交信息格式**：
```
<type>(<scope>): <subject>

<body>

<footer>
```

**Type 类型**：
- `feat`: 新功能
- `fix`: 修复 Bug
- `docs`: 文档更新
- `style`: 代码格式调整（不影响功能）
- `refactor`: 重构代码
- `perf`: 性能优化
- `test`: 测试相关
- `chore`: 构建/工具相关

**示例**：
```
feat(timer): 添加番茄钟暂停功能

- 添加暂停/继续按钮
- 暂停时保存当前剩余时间
- 继续时恢复计时状态

Closes #123
```

### 3. 人工审查

审查者按照以下清单进行审查：

#### ✅ 代码质量
- [ ] 命名规范，语义清晰
- [ ] 函数单一职责，不过长（< 50 行）
- [ ] 避免重复代码
- [ ] 错误处理完善（try-catch、边界检查）

#### ✅ 性能
- [ ] 避免频繁的 DOM 操作
- [ ] 缓存 DOM 查询结果
- [ ] 使用事件委托
- [ ] 避免内存泄漏（及时清理定时器、事件监听）

#### ✅ 安全性
- [ ] 防止 XSS 攻击（输入验证、输出转义）
- [ ] 不暴露敏感信息（GitHub Token 等）
- [ ] 使用 HTTPS 请求

#### ✅ 浏览器兼容性
- [ ] 支持 Chrome、Firefox、Safari、Edge 主流版本
- [ ] 不使用未广泛支持的 API
- [ ] 必要时添加 polyfill

#### ✅ 可维护性
- [ ] 代码结构清晰，易于理解
- [ ] 模块化良好，职责分离
- [ ] 易于扩展，不过度设计
- [ ] 注释恰当，不过度注释

### 4. 反馈修改

- 审查者使用建设性的语言提出建议
- 开发者及时响应，进行修改
- 重大问题需要重新审查

**反馈示例**：
> 建议：这里的循环可以优化，使用 `Array.find()` 代替 `for` 循环，代码更简洁。

### 5. 验证合并

- 所有审查意见已处理
- 代码通过 ESLint 检查
- 功能测试通过
- 合并到目标分支

## 📝 审查重点

### 高优先级（必须修复）
- ❌ 安全漏洞（XSS、CSRF、数据泄露）
- ❌ 功能性 Bug
- ❌ 性能严重问题（页面卡顿、内存泄漏）
- ❌ 代码无法运行

### 中优先级（建议修复）
- ⚠️ 命名不规范
- ⚠️ 代码重复
- ⚠️ 缺少错误处理
- ⚠️ 注释不足

### 低优先级（可选优化）
- 💡 代码风格不一致
- 💡 可以更简洁的写法
- 💡 性能微小优化

## 🎯 特定场景检查

### JavaScript 代码

- [ ] 使用 `const`/`let`，避免 `var`
- [ ] 使用箭头函数（回调函数）
- [ ] 使用模板字符串
- [ ] 使用解构赋值
- [ ] 使用 async/await 处理异步

### CSS 代码

- [ ] 使用 BEM 命名规范
- [ ] 使用 Flexbox/Grid 布局
- [ ] 使用 CSS 变量管理主题色
- [ ] 移动优先响应式设计

### HTML 代码

- [ ] 使用语义化标签
- [ ] 添加适当的 `alt` 属性
- [ ] 合理使用 `aria-*` 属性提升可访问性
- [ ] Meta 标签完整（viewport、charset）
